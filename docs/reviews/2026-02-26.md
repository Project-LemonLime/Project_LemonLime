## Executive Summary

`src/core/` 是 Project LemonLime 的评测核心，整体**功能完整、可运行**，但在架构演进过程中积累了显著的技术债。主要风险集中在：

1. **资源管理**：裸指针 `new`/`delete` 手动管理，无 RAII
2. **巨型文件**：`judgingthread.cpp` 单文件 1,402 行，混合三大职责
3. **数据模型**：`Contestant` 平行数组反模式，数据耦合脆弱

整体健康度：**B-**

## 架构概览

```
Contest (QObject)
├── Settings*
├── QList<Task*>
│   └── Task (QObject)
│       ├── QList<TestCase*>  (TestCase 非 QObject)
│       └── TaskType: Traditional | AnswersOnly | Interaction | Communication
└── QMap<QString, Contestant*>
    └── Contestant (QObject)
        └── 10 个平行 QList 存储每题编译状态、评测结果、得分、时间、内存
```

**评测流程**：
1. `Contest::judge()` 创建 `TaskJudger` 并提交给 `JudgingController`
2. `JudgingController` 管理线程池（`QThread` + `QQueue`），按 `maxJudgingThreads` 并行
3. `TaskJudger` 编译源代码，为每个测试点创建 `JudgingThread`（同步 `start()`/`wait()`）
4. `JudgingThread` 继承 `QThread`，负责运行程序并对比输出
5. 结果通过信号链逐级回传至 UI

**模块耦合**：`TaskJudger` 是耦合中心，同时依赖 `Contestant`、`Task`、`TestCase`、`Settings`、`JudgingThread`，违反 SRP。

## 1. 架构设计与模式

### 1.1 Contestant — 平行数组反模式

`contestant.h:56-68`：以 **10 个并行 QList** 存储每道题目数据：

```cpp
QList<bool> checkJudged;
QList<CompileState> compileState;
QStringList sourceFile;
QStringList compileMesaage;          // 拼写错误（见 §4）
QList<QList<QStringList>> inputFiles;
QList<QList<QList<ResultState>>> result;  // 三层嵌套！
QList<QList<QList<int>>> score;
QList<QList<QList<int>>> timeUsed;
QList<QList<QList<int>>> memoryUsed;
QDateTime judgingTime;
```

操作时须保持 10 个列表索引同步（`addTask`/`deleteTask`/`swapTask`），极易出错。
**建议**：引入 `TaskResult` 结构体，简化为 `QList<TaskResult>`。

### 1.2 TestCase 子任务依赖序列化 Hack

`testcase.cpp:113-114`：子任务依赖信息被编码为带特殊后缀的文件名，混入 `inputFiles` 数组序列化：

```cpp
inputFiles.push_back(QString("%1_lemon_SUbtaskDEPENDENCE_fLAg").arg(i));
```

若用户输入文件名恰好以 `_lemon_SUbtaskDEPENDENCE_fLAg` 结尾，数据将被损坏。
**建议**：在 JSON 中使用独立的 `dependenceSubtask` 字段。

## 2. 鲁棒性与错误处理

### 2.1 Contestant 数组访问无边界检查

`contestant.cpp:21-39`：所有 getter/setter 直接使用 `[]` 索引，无边界检查：

```cpp
auto Contestant::getCheckJudged(int index) const -> bool { return checkJudged[index]; }
// 所有 getter/setter 同理
```

无效 `index` 导致未定义行为（越界读/写）。
**建议**：添加 `Q_ASSERT(index >= 0 && index < checkJudged.size())`。

### 2.2 `readFromStream` 缺乏校验

所有 `readFromStream(QDataStream&)` 对输入数据不做合法性校验——`count` 可为负数或极大值，直接用于控制循环。恶意构造的文件可能导致 OOM 或崩溃。

## 3. 性能与效率

### 3.1 Busy-Wait 轮询

特判进程、编译器进程等待均使用 `while + msleep(10)` 轮询（`judgingthread.cpp:501,611` 等多处）。可改用 `QProcess::waitForFinished(timeout)` 或 `finished` 信号提升效率。

### 3.2 不必要的中间拷贝

`task.cpp:291`、`judgingcontroller.cpp:69`：

```cpp
for (auto [x, y] : this->compilerConfiguration.toStdMap()) { ... }
```

`toStdMap()` 创建整个 map 的深拷贝。应直接使用 map 的迭代器遍历。

### 3.3 `checkDependenceSubtask` O(n²) 重复检测

`testcase.cpp:84-86`：双重循环检测重复值。可改用 `QSet` 优化为 O(n)。

### 4.1 巨型文件

### 4.3 信号参数缺乏语义

`contest.h:74`：


7 个匿名 `int` 参数，调用者和接收者均难以理解每个参数含义。
**建议**：定义 `SingleCaseResult` 结构体作为信号参数。

### 4.4 `BufferedStreamReader` 错误信息截断

`judgingthread.cpp:185,204`：`nextUntilNewLine()` / `nextUntilSpace()` 每次最多读 32 字符（分块比较，评测正确性不受影响），但错误消息 `Read "%1" but expect "%2"` 仅展示当前分块的 32 字符片段，而非完整行内容。

## 5. 安全性

### 5.1 特判程序路径未限制

`judgingthread.cpp:481`：

```cpp
judge->start(Settings::dataPath() + task->getSpecialJudge(), arguments);
```

`task->getSpecialJudge()` 来自 JSON 反序列化，未做路径遍历检查。恶意构造的比赛文件可将路径设为 `../../malicious_binary` 执行任意程序。
**建议**：校验 `specialJudge` 不包含 `..` 或绝对路径，限制在 `dataPath()` 之下。
